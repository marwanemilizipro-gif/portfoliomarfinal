Déploiement Système

Déploiement Automatisé d'Images OS

Serveur FOG Project · Stage 2ème année · Lycée Jacques Brel

Contexte

Le lycée avait besoin d'une solution pour réinstaller rapidement les salles C207 et C208 avec une image système propre (Windows 11, Linux Mint ou Ubuntu) via le réseau. Le projet s'est déroulé en deux phases : maquettage dans un réseau isolé sur Proxmox (192.168.2.x) derrière IPFire, puis migration en production sur le réseau administrateur du lycée.

Architecture & Installation

1

Serveur FOG déployé sur une VM Debian 13 (Trixie) hébergée sur le cluster Proxmox du lycée, avec adresse IP fixe 172.28.30.15.

2

Le DHCP est géré par le contrôleur de domaine Windows (Active Directory) — pas de DHCP FOG activé pour éviter les conflits réseau.

3

Extension du stockage à 282 Go via GParted à chaud pour accueillir toutes les images système.

Configuration PXE

1

Configuration des options DHCP sur le serveur Windows AD : option 66 (Boot Server) pointant vers l'IP du serveur FOG.

2

Option 67 (Boot File) : ipxe.efi pour les postes UEFI récents et undionly.kpxe pour la compatibilité Legacy/BIOS.

3

Désactivation du Secure Boot et de BitLocker sur les postes clients avant capture.

Masterisation des images

1

Image Windows 11 créée avec procédure Sysprep (sysprep /generalize /oobe) pour supprimer les SID et éviter les conflits lors du déploiement multi-postes.

2

Capture en mode "Single Disk – Resizable" pour compatibilité avec différentes tailles de disques.

3

Images Linux Mint et Ubuntu configurées avec les logiciels open-source des TP, puis capturées et testées.

4

Déploiement en production sur les salles C207 et C208 — postes opérationnels en quelques minutes par machine.

Compétences mobilisées

FOG Project PXE Boot UEFI/Legacy Debian 13 Sysprep Windows 11 Linux Mint Ubuntu Proxmox Active Directory GParted

Résultat

Serveur FOG opérationnel en production au lycée. Déploiement des salles C207 et C208 en quelques minutes par poste. Trois images disponibles : Windows 11, Linux Mint et Ubuntu, déployables automatiquement via PXE sans intervention manuelle.

Veille Technologique

La Conteneurisation

Docker & Kubernetes – Apports pour l'infrastructure et défis de sécurité

Problématique

"Comment la conteneurisation avec Docker et Kubernetes transforme-t-elle les infrastructures modernes, et quels nouveaux défis de sécurité introduit-elle ?"

Pourquoi ce choix ?

L'industrie IT délaisse progressivement les machines virtuelles traditionnelles au profit des conteneurs, plus légers et plus rapides. Un conteneur Docker démarre en quelques secondes là où une VM nécessite plusieurs minutes. Cette révolution redéfinit le déploiement et la gestion des infrastructures modernes — directement en lien avec mon orientation SISR.

Ma démarche de veille

Je suis activement cette technologie via Feedly pour agréger les flux RSS, LeMagIT pour les actualités IT en français, et les documentations officielles Docker et Kubernetes. Je me concentre particulièrement sur les usages en infrastructure, la sécurité des conteneurs et l'orchestration à grande échelle.

VM vs Conteneur — La différence fondamentale
🖥️ Machine Virtuelle
Simule un PC complet avec son propre noyau OS.
Lourde (plusieurs Go), démarre en plusieurs minutes.
Très isolée mais consomme beaucoup de ressources.
Ex : mes VMs sur Proxmox au lycée
📦 Conteneur Docker
Partage le noyau Linux de la machine hôte.
Léger (quelques Mo), démarre en quelques secondes.
Embarque uniquement l'appli et ses dépendances.
Portable : tourne pareil partout

💡 Analogie : La VM c'est comme louer un appartement entier. Le conteneur, c'est louer une chambre dans une colocation — tu partages les communs (le noyau) mais tu as ton espace propre (l'appli).

🐳 Docker — Les concepts clés
IMAGE
Le modèle

Comme une image Sysprep dans FOG — un modèle en lecture seule contenant tout ce qu'il faut pour faire tourner l'application.

CONTENEUR
L'instance qui tourne

C'est l'image qui s'exécute. Comme un poste déployé depuis FOG — le modèle devient une machine réelle.

DOCKERFILE
La recette

Le fichier texte qui décrit comment construire une image. Comme un script d'installation automatisé.

💡 Problème résolu : "Ça marche sur mon PC mais pas sur le serveur" — Docker élimine ce problème en empaquetant l'appli avec TOUT ce dont elle a besoin. Même environnement partout.

☸️ Kubernetes — L'orchestration à grande échelle

Docker lance un conteneur. Mais en entreprise, il faut en gérer des centaines sur plusieurs serveurs. Kubernetes (K8s) est le chef d'orchestre : il décide où lancer les conteneurs, les redémarre automatiquement s'ils plantent, et répartit la charge.

Concepts fondamentaux
Cluster → ensemble des serveurs gérés par K8s
Node → un serveur du cluster (Master ou Worker)
Pod → la plus petite unité : 1 ou plusieurs conteneurs
Deployment → "je veux 3 copies de cette appli"
Service → répartit le trafic entre les pods (comme HAProxy)
Ce que K8s fait automatiquement
✓ Redémarre un conteneur planté
✓ Répartit les conteneurs sur les serveurs disponibles
✓ Met à jour sans interruption de service
✓ Adapte le nombre de conteneurs à la charge
✓ Gère les accès réseau entre conteneurs

💡 Analogie : Docker = un ouvrier qui construit. Kubernetes = le chef de chantier qui coordonne 100 ouvriers sur 10 chantiers simultanément, et rappelle un remplaçant si quelqu'un tombe malade.

🔗 Lien avec mon parcours SISR
Ce que j'ai fait → Ce que K8s fait mieux
VMs sur Proxmox → Conteneurs Docker (plus légers)
HAProxy manuel → Service K8s (automatique)
Supervision Centreon → Health checks K8s (intégrés)
Déploiement FOG → Déploiement K8s (en secondes)
Pourquoi c'est important pour le SISR
Les datacenters migrent vers les conteneurs
Haute disponibilité automatique sans intervention
Sécurité : isolation des applis, gestion des CVE
Compétence clé pour un technicien infrastructure
🔐 Les risques de sécurité — Ce que ça change
⚠️ Images malveillantes

Docker Hub est public — n'importe qui peut y uploader une image. Certaines contiennent des logiciels malveillants cachés qui volent des données ou utilisent le serveur pour miner de la cryptomonnaie.

Solution : utiliser uniquement des images officielles et les scanner avec des outils comme Trivy.

⚠️ Noyau partagé

Tous les conteneurs partagent le noyau du serveur hôte. Si un attaquant exploite une faille dans ce noyau, il peut accéder à tous les conteneurs. Avec une VM, chacune a son propre noyau — isolation totale.

Solution : mettre à jour régulièrement le système hôte.

⚠️ Droits trop élevés

Par défaut beaucoup de conteneurs tournent en mode administrateur. Si un attaquant prend le contrôle du conteneur, il a les droits admin — comme si tous tes utilisateurs AD étaient administrateurs.

Solution : principe du moindre privilège — même logique que les GPO en Active Directory.

⚠️ Réseau non cloisonné

Par défaut dans Kubernetes, tous les conteneurs peuvent se parler librement. Un conteneur compromis peut donc attaquer tous les autres.

Solution : Network Policies K8s — même principe qu'IPFire entre mes zones LAN/DMZ/EXT.

✅ Dans mon infra : Graylog centralise les logs de tous les équipements, Wazuh détecte les comportements suspects, IPFire cloisonne les zones réseau. Ces mêmes principes s'appliquent directement à la sécurisation d'un environnement Docker/Kubernetes.
